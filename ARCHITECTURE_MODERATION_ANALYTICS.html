Category 2

Architecture Styles

Hai service thể hiện rõ phong cách microservices, event-driven architecture và xử lý bất đồng bộ bằng RabbitMQ. Một số mục thuộc rubric như sơ đồ C4 cần được trình bày trong báo cáo/slide; file này đã bổ sung sơ đồ dạng HTML để làm minh chứng.

Advantages / disadvantages Đạt
  • Ưu điểm: loose coupling, scale consumer độc lập, retry/DLQ.
  • Nhược điểm: eventual consistency, cần quản lý schema event và DLQ.
 Có thể trình bày trực tiếp phần ưu/nhược điểm này trong báo cáo.
Compare with other architectures Đạt
  • So với monolith: tách moderation và analytic giúp giảm coupling và deploy độc lập.
  • So với synchronous microservices: RabbitMQ giảm block request realtime.
 Lý do chọn EDA hợp lý vì chat/media cần latency thấp.
Trade-off Đạt
  • Hiệu năng: Aho-Corasick in-memory quét text nhanh.
  • Chi phí: AWS Rekognition chỉ dùng cho ảnh, không gọi cho text.
  • Độ phức tạp: cần DLQ, idempotency, event contract.
 Trade-off giữa latency realtime và độ phức tạp vận hành là rõ.
Contextual questions Đạt
  • Traffic tăng: scale thêm consumer moderation/analytic theo queue backlog.
  • Service lỗi: retry 3 lần, message lỗi đi DLQ.
  • User-service lỗi khi enrich: analytic fallback, không làm sập dashboard.
 Có câu trả lời thực tế cho traffic, downtime, scaling, fault tolerance.

Style 1 - Microservices

Moderation và Analytic là hai bounded context riêng, có Dockerfile, cấu hình runtime và database riêng. Điều này giúp scale và deploy độc lập.

Style 2 - Event-Driven

RabbitMQ là trục giao tiếp chính. Moderation nhận request qua queue, publish violation event; Analytic consume các event để ghi log/dashboard.

Style 3 - Layered Service

Hai service đều chia lớp controller/service/repository/entity/config. Moderation còn có routing/provider layer cho thuật toán kiểm duyệt.

Category 3

Architecture Characteristics

Nhóm này được chấm theo các đặc tính vận hành: availability, performance, fault tolerance, security, scalability và maintainability. Một vài tiêu chí như Redis cache/rate limiter chưa có minh chứng trong riêng hai service này.

Đạt một phần
Tiêu chí Đánh giá Minh chứng trong code Ghi chú cần nói khi bảo vệ
Availability 24/7 Một phần
  • docker-compose.prod.yml: container restart: unless-stopped.
  • RabbitMQ healthcheck và durable queues.
  • DLQ giúp không mất message lỗi.
 Chưa thấy multi-AZ/load balancer/HPA cho riêng hai service, nên không nên nhận là 24/7 hoàn chỉnh.
Redis CRUD object / cache Chưa đạt
  • Compose có Redis ở cấp hệ thống.
  • Chưa thấy moderation-service hoặc analytic-service dùng Redis trực tiếp.
 Nếu muốn ăn điểm này cần thêm cache rule dictionary metadata, dashboard summary hoặc distributed lock.
Rate limiter client Chưa đạt
  • Không thấy logic rate limit ở frontend/admin client cho các API moderation/analytic.
 Không nên claim điểm này nếu chỉ tính hai service và UI hiện tại.
Retry 3-5s Đạt
  • moderation-service/application.yml: retry enabled, max-attempts 3, max-interval 5000.
  • analytic-service/application.properties: retry enabled, max-attempts 3, max-interval 5000.
 Đây là retry cho RabbitMQ listener, phù hợp tiêu chí fault tolerance async.
Rate limiter server/gateway Chưa đạt
  • Không thấy RequestRateLimiter hoặc cấu hình rate limiter trong hai service.
  • Gateway có circuit breaker, nhưng circuit breaker không phải rate limiter.
 Cần bổ sung gateway rate limiter bằng RedisRateLimiter nếu muốn lấy điểm này.
JWT security Một phần
  • analytic-service/SecurityConfig.java: JWT Resource Server HS512.
  • AdminAnalyticsController.java và ModerationAnalyticsController.java: @PreAuthorize("hasRole('ADMIN')").
  • Moderation rule controller chưa thấy security annotation/service-level JWT riêng.
 Analytic đạt rõ; moderation có thể được bảo vệ qua API Gateway, nhưng trong service code chưa đủ minh chứng.
Scalability Một phần
  • RabbitMQ queue cho phép scale horizontal bằng nhiều consumer instance.
  • Service stateless tương đối; state nằm ở PostgreSQL/RabbitMQ.
  • Compose hiện chỉ chạy 1 container mỗi service.
 Kiến trúc hỗ trợ scale, nhưng chưa có cấu hình replica/load balancer trong repo.
Maintainability Đạt
  • Moderation: controller, service, repository, entity, contracts, provider, routing.
  • Analytic: controller, service, repository, entity, listener, dto, config.
  • Có defensive validation trong router/service/listener.
 Cấu trúc module rõ, dễ bảo trì và mở rộng provider/event mới.
